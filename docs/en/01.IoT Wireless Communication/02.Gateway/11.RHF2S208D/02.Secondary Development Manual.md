# Secondary Development Manual

## Introduction

The RHF2S208D allows users to perform secondary development. The RHF2S208D's software is developed based on the Raspberry Pi operating system, and on this basis, we have developed a rich set of easy-to-use software interfaces. Furthermore, we provide support for users to integrate third-party LoRa services.

This article will not only introduce these software interfaces and how to use them, but also guide you on how to integrate third-party LoRa services more  quickly through the support we provide.

## Hardware resources

The hardware resources of the RHF2S208D baseboard are as follows:

*   1 Ethernet interface
*   2 SX1302 LoRa modules
*   1 4G module
*   1 WiFi module
*   1 GPS module
*   1 SIM card slot
*   1 eSIM patch slot
*   1 lithium battery charging indicator light
*   1 solar cell charging indicator light
*   1 power indicator light
*   1 LTE indicator light
*   1 RS485 serial port
*   1 serial port for development

The RHF2S208D uses a Raspberry Pi CM4 as its main controller, and its hardware resources are as follows:

*   Main chip: Broadcom BCM2711
    
    *   Architecture: ARM Cortex-A72 64 位
    *   Clock speed: 1.5GHz
    *   Number of cores: 4
    *   Memory: DDR4 2GB
    *   eMMC: 32GB

## Software Resources

- Based on Linux kernel 6.12.47+rpt-rpi-v8;
- SPI driver;
- I2C driver;
- USB Host/Device driver;
- LoRaWAN module driver;
- 4G modem driver, supporting WCDMA/TD-LTE/GPRS/EDGE data transmission;
- GPS driver, supports precise GPS time synchronization;
- Ethernet driver;
- WiFi/BT driver;
- DMA driver;
- LED driver;
- Power management driver;
- RS485 serial port driver;
- Data collected by an internal temperature sensor;
- Internal hardware watchdog;
- Bootloader;
- Supports image flashing;

## Debugging interface

The RHF2S208D offers a variety of debugging methods:

*   Web interface
*   serial port debugging
*   SSH login

Regardless of whether you log in to the device via serial port or SSH, the default username and password are as follows:

Username: rxhf

Password: risinghf

### Serial port debugging

Before connecting your device and computer, please install the FTDT USB to UART driver. Open the device cover and use a USB to serial adapter to connect the device's serial port to your computer's USB port. You will then see a COM port on your computer. In ExtraPutty, select "Serial" and enter the correct COM port number and baud rate (115200). You should then be able to successfully connect to the device.

![image-20230216184550055](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/9f07832acf3a9318caa97060a4fd69d5.png)

Then enter the username "rxhf" and password "risinghf" to log in to the device.

![image-20230216184624937](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/9c6d9b693e790da58aa560bae29ce5d8.png)

### SSH login

#### Login using LAN DHCP IP address

Connect the device to the local area network (LAN) via an Ethernet cable, and use the DHCP service of the upstream router to find the device's IP address on the LAN. After obtaining the device's IP address, use the PuTTY tool to log in to the command terminal.

![image-20230216185227760](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/0f68889ea38831c99268da6a50140eb9.png)

#### Log in using WiFi IP address

Scan for WiFi signals on your computer, find the device with SSID RHF2S208D_XXXXXX, and connect. After connecting, the computer's default IP address is 192.168.8.1. Once you have the device's IP address, use PuTTY to log in to the command terminal. The default WiFi SSID and password are as follows:

SSID: RHF2S208D_XXXXXX

Password: RisingHF_XXXXXX

Note: "XXXXXX" ends with the last 3 bytes of the device's MAC address. If there are letters, use uppercase letters.

![image-20230216185103177](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/fbee7ae126865e43e506a0205f2c455c.png)

## Setting up a cross-compilation environment

The RHF2S208D currently integrates LoRaserver, IoTSquare, and our company's self-developed LoRa service program, which is compatible with the standard packet_forwarder. Users can choose to start one of the LoRa service programs to connect to the corresponding server, or develop a new LoRa service program and integrate it into the gateway to connect to third-party target servers. The gateway provides a cross-compilation method to help users build their own binaries for device operation.

**This tutorial describes how to set up a cross-compilation environment based on the Ubuntu 22.04.5 LTS 64-bit operating system.**

Cross-compile test.c

```
PC $> make
```

Upload to device operation

```
$> ./test
hello world!
```

### Connect to your server

Packet Forwarder is a packet forwarding application that acts as a bridge between nodes and servers. On one hand, it receives LoRa protocol data sent by nodes and processes the data, including data parsing, verification, and format conversion. On the other hand, it uses a network connection to transmit the processed data packets to the server. For example, the lora_pkt_fwd compiled from the official source code transmits data packets to the server via the UDP network protocol. Therefore, we need to specify the IP address of the server that Packet Forwarder needs to communicate with.

Assuming your server's IP address is 192.168.0.12, you need to modify the global_conf.json configuration file as shown in the image below:

![image](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/RHF2S025B/user_manual/71a69f98-d7d4-48d6-a860-3676ebc5ee90.png)

Of course, you can also create a local_conf.json file and add a server_address configuration item to override the same configuration item in global_conf.json:

![image](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/RHF2S025B/user_manual/8c7f7297-0c29-4955-8dc1-bff7c16d9b37.png)

Other configuration items should be changed according to the actual situation, such as the unique identifier gateway_ID, and the port numbers for the uplink and downlink processes on the server, ser_port_up and ser_port_down, etc.

### Use lora_service to manage LoRa services

In the previous section, we successfully connected to and started the specified server using the official Packet Forwarder. In this section, we will introduce how to use lora_service to more conveniently manage newly added LoRa services.

#### Disable built-in LoRa service

The RHF2S208D supports multiple LoRa services, such as RisingHF, which is based on Packet Forwarder and uses a customized UDP protocol, and a direct Packet Forwarder Link. There's also IoTSquare, which connects to an MQTT server via a bridge. The `lora_service` application manages these LoRa services. It is recommended that newly added LoRa services also be registered with `lora_service` for unified management.

Of course, you can choose not to use it.

```
$> sudo lora_service -s
```

#### Add system service

To start the process using systemd's auto-start service, add the file `foo-pktfwd.service` to the `/lib/systemd/system` directory . Note: If you configure the service as shown below, please remove the `#` symbol and the comment after it; otherwise, it may not work correctly.

ExecStartPre=/usr/local/sbin/gwrst # This process is executed before the final process starts.
ExecStartPre=/usr/local/sbin/gwrst1 # This process is executed before the final process starts.
ExecStart=/opt/pktfwd/lora_pkt_fwd # The process that is eventually started.

[Install]
WantedBy=multi-user.target 
```

The above briefly describes how to start a LoRa service process. As you'll see in foo-pktfwd.service, the reason the command `/usr/local/lora/lora_service -ffoo-pktfwd` must be executed before starting the final process is because this command serves to prevent other processes from starting. The `-f` parameter means that all other processes (those registered with `lora_service`) except for the foo-pktfwd process must stop.

#### Register for LoRa service

Execute the following command to register the LoRa service with the lora_service manager.

    $> sudo lora_service -a '{"foo-pktfwd":["foo-pktfwd.service"]}'

After registering LoRa service mutual exclusion, your process will be mutually exclusive with other LoRa service processes. Only one LoRa service can be started at a time to prevent LoRa module conflicts.

#### Registration web interface menu

Register a web interface LoRa service menu so that users can start or stop the LoRa service through the web interface.

```

--name: Web interface name, a unique identifier.

--menu: The title of the web interface menu.

--dif: Optional frequency configuration file directory.

After executing the command, foo-pktfwd will start immediately, and the LoRa service will also start automatically when the RHF2S208D boots into the system.

### lora_service

The gateway integrates numerous LoRa service processes, but LoRa only allows one process to occupy it at a time. In other words, all LoRa-related processes are mutually exclusive, but there is no connection between the processes themselves. How can users avoid this problem?

The gateway encapsulates a lora_service process based on systemd. Users can use systemd to manage the auto-start of their own LoRa-related processes and register the service with lora_service. In this way, the process can be mutually exclusive with other LoRa processes.

he following explanation uses the standard packet_forwarder as an example. The packet_forwarder program is located in /opt/pktfwd, which is also its working directory. The normal way to start it is to execute ./lora_pkt_fwd . However, if other related LoRa service processes are running at the same time, unexpected problems such as startup failure or inability to receive data may occur.

Therefore, it is recommended to use systemd's auto-start service to start the process, which is achieved by adding a file named xxx.service (where xxx is a user-selectable name, but cannot be duplicated with other files) to the /lib/systemd/system directory. The following explains how to implement the service startup method for the standard packet_forwarder process.

```
$> cat /lib/systemd/system/pktfwd.service
[Unit]
Description=Semtech LoRa packet forwarder # This server description information
Wants=network-online.target
After=network-online.target # Start this service after the network service.

[Service]
Restart=always # Action to restart the process after it exits.
RestartSec=5 # Used in conjunction with Restart to restart the process 5 seconds after it exits.
WorkingDirectory=/opt/pktfwd/ # Working directory of the process
ExecStartPre=/usr/local/sbin/web_select_lora_freq --name pktfwd # Set the web interface state before starting the final process
ExecStartPre=/usr/local/sbin/lora_service -fpktfwd # Before starting the final process, mutually exclude other LoRa service processes.
ExecStartPre=/usr/local/sbin/gwrst # This process is executed before the final process starts.
ExecStartPre=/usr/local/sbin/gwrst1 # This process is executed before the final process starts.
ExecStart=/opt/pktfwd/lora_pkt_fwd # The process that is eventually started

[Install]
WantedBy=multi-user.target 
```

The above briefly describes how a LoRa service process starts. As seen in pktfwd.service, the reason the command `/usr/local/lora/lora_service -fpktfwd` must be executed before starting the final process is because this command serves to prevent other processes from starting. The `-f` parameter means that all other processes (those registered with `lora_service`) except for the pktfwd process must stop.

You can use `lora_service -h` to view the relevant usage instructions.

```
$> lora_service -h

Available options:
 -h                   print this help
 -c <file>            load config file
 -a <str>             add(replace) lora service
 -d <str>             del lora service
 -l                   list lora service
 -q                   query use lora service
 -s [str]             stop lora service
 -f [str]             filter stop lora service
 -t [str]             temporary stop lora service
 -r <str>             restart/start lora service
 -T <str>             temporary restart/start lora service

Example:
 lora_service -a '{"service_name":["service1.service","service2.service",...]}'
 lora_service -d service_name
 lora_service -l
 lora_service -q
 lora_service -s[service_name]

```

### Noise floor scan

The device features a built-in noise floor analysis engine, which helps users scan and analyze environmental noise floor on-site, enabling early detection of interfering signals and noise during customer deployment. This allows for network optimization and avoidance of strong interference channel frequencies. **Note: LoRa service must be disabled before using noise floor scanning.**

The noise floor scanning function is very easy to operate; simply enter the start and end frequencies in the command parameters. The default step size is 200,000 Hz, the number of samples is 2,000, and the rssi_offset compensation is -20dB for the RHF2S208DxHx-470 series gateways, -10dB for the RHF2S208DxHx-868 series gateways, -20dB for the RHF2S208DxHx-915 series gateways, and -26dB for the RHF2S208DxFx-470 series gateways. The wider the scanning bandwidth, the longer the time required. It is generally recommended to scan only 2MHz or 5MHz to cover the target operating frequency band. **Note: Due to hardware limitations, the RHF2S208Dxxx-470 series can only scan 470~490MHz, the RHF2S208Dxxx-868 series can only scan 862~876MHz, and the RHF2S208Dxxx-915 series can only scan 902~928MHz.**

Ambient noise floor is a crucial parameter affecting wireless communication. When setting up and installing a gateway in the field, a preliminary analysis of the ambient noise level is essential to prevent high noise or strong interference from impacting communication distance and reliability. While we aim for the lowest possible ambient noise, completely noise-free environments are unrealistic.
An ambient noise floor of around -110dBm is generally considered ideal. A maximum noise level of -100dBm is also permissible; otherwise, communication distance will be severely limited. If the noise level exceeds -95dBm, it is strongly recommended to switch operating frequencies to avoid interference.

### The system only executes once during startup

Scripts in the `/etc/initonce.d/` directory are executed only once at boot. If execution is successful (the command returns 0), the system will delete the command and it will not be executed again on the next boot. If execution fails (the command returns a non-zero value), the system will still execute it on the next startup. This configuration is suitable for users who want their systems to execute only once at boot.

### Network Status Reporting Interface

When the routing path of the backhaul network changes, the system executes all commands in the `/etc/route- call.d` directory. This directory exposes two variables, `ROUTE_BEFOR` and `ROUTE_CURR` , representing the previous and current routing paths, respectively. This is useful for users to monitor whether network routing paths have changed.

### External power status reporting interface

When the external power supply status changes, the system executes all commands in the `/etc/power- call.d/ ` directory. This directory exposes two variables, `NAME` and `STATE` , representing the external power supply name and current status, respectively. This is suitable for users to monitor whether the external power supply status has changed.

### Report backup lithium battery status interface

When the backup lithium battery's state changes, the system executes all commands in the `/etc/battery- call.d/ ` directory. This directory exposes two variables, `BAT_BEFOR` and `BAT_CURR` , representing the backup lithium battery's previous and current states, respectively. This is useful for users to monitor whether the backup lithium battery's state has changed.

Because backup lithium batteries use a polling method to capture state changes, all commands in the `/etc/battery- call.d/` directory are executed every time the system boots up. Furthermore, if the time interval between state changes is short, the current state record may be lost. External power supplies use interrupts to capture state changes, thus avoiding this issue.

## Device logs

The RHF2S208D has two log output directories: /var/log and /var/rlog. The main difference between these two directories lies in whether the logs are cleared after a device reboot. The /var/log directory primarily stores system logs, which are cleared after a device reboot; these are commonly known as non-persistent logs and store a large amount of output logs that are not particularly important. The /var/rlog directory primarily stores logs related to critical processes; these are unaffected by device reboots and are commonly known as persistent logs. These store a small amount of output logs that are particularly important and convenient for future monitoring of the device's status.

### /var/log (non-persistent log)

Non-persistent logs are stored up to 100MB. Once the size exceeds 100MB, no more logs can be written, but this will not affect program usage. The device will periodically shuffle the logs to ensure that the non-persistent logs do not exceed 100MB.

/var/log/kern.log records the kernel's runtime logs, including kernel status information, which users can use to analyze the operating status of the device system.

#### syslog log

/var/log/syslog records kernel logs, systemd logs, and process standard error output, covering almost all system logs.

#### auth log

/var/log/auth.log records the logs of users logging into devices.

#### lighttpd log

## troubleshooting

### LoRa service process conflict

LoRa modules are dedicated resources; a gateway can only have one LoRa module available for use by a single application. If multiple applications use the LoRa module simultaneously, unpredictable problems may occur. Therefore, during use and testing, it is crucial to ensure that the LoRa service module is run by only one application.

Currently, the gateway system integrates the following programs to start the LoRa service:

- packet_forwarder
- iotsquare
- loraserver
- LoRa testing program
- background noise scan

#### ps

Use the command `ps -ef | grep -E "lora_pkt_fwd|pktfwd|loragw|spectral_scan"` to check if there are LoRa service-related programs on the system.

- lora_pkt_fwd is a keyword for the packet_forwarder process.
- pktfwd is a keyword used in the iotsquare and loraserver processes.
- loragw is a keyword in the LoRa testing process.
- spectral_scan is the process keyword for noise floor scanning.

If the LoRa module is occupied, two or more records will be returned (one of which is a record of its own command). The user can determine which process is executing the system based on the returned process keyword. As shown below, the LoRa module is being occupied by the iotsquare service (please refer to the device's user manual for details on how to disable the service).

```
$> ps -ef | grep -E "lora_pkt_fwd|pktfwd|loragw|spectral_scan"
root      103607       1  2 09:00 ?        00:00:00 /opt/rhf-bridge/pktfwd # LoRa modules are being used by the iotsquare service.
rxhf      103658    5107  0 09:00 pts/0    00:00:00 grep --color=auto -E lora_pkt_fwd|pktfwd|loragw|spectral_scan
```

#### lora_service

The gateway has a custom `lora_service` command that manages the gateway's LoRa service process, ensuring that only one LoRa service process runs at a time. You can also use this command to query or shut down running LoRa processes.

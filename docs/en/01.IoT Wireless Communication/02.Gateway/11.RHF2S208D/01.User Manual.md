# RHF2S208D User Manual

## Foreword

The RHF2S208D is a high-performance IoT LoRaWAN gateway designed and developed independently by Ruexing Hengfang Networks (Shenzhen) Co., Ltd., primarily for outdoor applications. Internally, it uses a Raspberry Pi CM4 as the main controller, with a 64-bit ARM Cortex-A72 architecture CPU, quad-core, and a clock speed of 1.5GHz, providing powerful data processing capabilities. In addition, the RHF2S208D is equipped with two SX1302 LoRa modules, one GPS module, one Ethernet interface, one LTE module, one WiFi module, one temperature sensor, and one RS485 serial port for solar panel data acquisition. It supports multiple power supply methods, including DC, PoE, lithium battery, and solar panel, to meet various installation scenarios.

This article will detail how to operate the gateway using the built-in web management platform of the RHF2S208D. Then, in conjunction with the LoRaWAN service provided by RisingHF -- IoTSquare and a node, it will demonstrate the actual use of the gateway by setting up a complete communication example. Next, it will introduce the method of upgrading the firmware. Finally, this article will provide some solutions to common problems, hoping to help users.

## Know Your Gateway

### Gateway supported functions

*   It possesses stable and reliable LoRaWAN communication capabilities, with a communication range covering multiple frequency bands centered at 470, 868, and 915 MHz. 
*   It supports both Ethernet and LTE backhaul networks.
*   WiFi supports AP mode for near-field wireless debugging.
*   Supports GPS atomic time synchronization with LoRaWAN. 
*   Supports parsing data from the solar cell's RS485 and uploading it to a cloud server. 
*   Includes a built-in, easy-to-use web gateway management platform.
*    Supports LoRaWAN 470 full/half dual-mode (optional). 
*   Supports LoRaWAN 16 channels (optional).

### Gateway external physical interface

The gateway has a total of 4 antenna ports, 1 external power input port, 1 RJ45 port, 1 RS485 serial port, and 1 debug serial port.

- ANT1: LoRa antenna port.
- ANT2: LoRa antenna port.
- ANT3: GPS antenna port.
- ANT4: LTE antenna port.
- ANT5: WiFi antenna port.
- Power: External power input interface, power input range DC12V to 28V.
- RJ45: RJ45 network port.
- RS485: Input information for connecting to the solar panel. 
- Debug serial port: Used for local debugging of the device.

### power supply

The RHF2S208D is powered by a standard AC/DC industrial power supply. All RHF2S208D gateways come standard with a high-reliability, industrial-grade IP67 waterproof AC/DC power supply. Additionally, it can be powered via PoE in a server room. Outdoors, it can also be powered by solar panels and batteries. For the RHF2S208DBxx series gateways, a built-in 3200mAh lithium iron phosphate rechargeable battery is also included as a backup power source. All power sources can be connected to the device, and the device intelligently selects the operating power source based on the priority order: AC/DC industrial power supply > PoE power supply > solar panel > backup battery.

### Backhaul network

The RHF2S208D supports two backhaul network modes: Ethernet and LTE.

Ethernet: Connecting the gateway's RJ45 port to the LAN port of the upstream router using a network cable enables a 10/100m Ethernet connection to the server.

LTE: Please open the device cover, insert the SIM card, and then power on the device to restart the system. If the device has an eSIM card soldered on, it will automatically dial upon power-on.

You can log in to the built-in web interface to configure the priority of the backhaul network.

### Debugging interface

The RHF2S208D provides a variety of debugging interfaces.

- Web interface

- serial port debugging
- SSH login

This document mainly introduces how to use the debugging interface in a web interface.

## Web management platform for gateways

The RHF2S208D's built-in web management platform provides users with an entry point to operate the gateway. It is very convenient and easy to use, allowing users to easily manage and operate their own gateway. This chapter will detail the various parts of the management platform and their usage.

### Get IP

- Open the web interface via local area network

Connect the device to the local area network (LAN) via an Ethernet cable, and use the DHCP service of the upstream router to find the device's IP address on the LAN. After obtaining the device's IP address, open the IP address in a browser to access the device's built-in web interface.

- Open the web interface via WiFi

Scan for WiFi signals on your computer, find the device with SSID RHF2S208D_XXXXXX, and connect. After connecting, the computer's default IP address is 192.168.8.1. Once you have the device's IP address, open 192.168.8.1 in your browser to access the device's built-in web interface. The default WiFi SSID and password are as follows:

SSID: RHF2S208D_XXXXXX

Password: RisingHF_XXXXXX

Note: "XXXXXX" ends with the last 3 bytes of the device's MAC address. If there are letters, use uppercase letters.

### Server

The server configuration involves LoRa-related services, including environmental noise floor scanning and connection configurations for various manufacturers' servers. Note: Only one LoRa service can be running at a time. If a user has already started one service, starting another server will cause the device to shut down the original service before starting the new one.

#### Ambient Noise Scan

The device has a built-in noise floor analysis engine that helps users scan and analyze environmental noise floor on-site, allowing them to detect interference signals and noise in advance during customer deployment. This enables network optimization and avoidance of channels and frequencies with strong interference.

The noise floor scan function is very easy to operate. Simply enter the start and end frequencies and click "Start Scan". If you only enter the start frequency, the default scan bandwidth is 2MHz and the default step is 200kHz. The wider the scan bandwidth, the longer it will take. It is generally recommended to scan only 2MHz or 5MHz to cover the target operating frequency band. Note: Due to hardware limitations, the RHF2S208Dxxx-470 series can only scan 470~490MHz, the RHF2S208Dxxx-868 series can only scan 862~876MHz, and the RHF2S208Dxxx-915 series can only scan 902~928MHz.

![image-20250616160516103](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/image-20250617100718056.png)

Fill in the start frequency, end frequency, and scan step, then click the "Start Scan" button on the right.

When the noise floor scan is complete, the configuration interface will display the scan results in two forms: the noise floor thermal distribution map represents the transient noise distribution, and the average noise distribution map represents the average noise level obtained at that frequency point during the scan.

![image-20250616160559750](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/image-20250617100750751.png)

![image-20250616160624200](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/image-20250617100805291.png)

Ambient noise floor is a crucial parameter affecting wireless communication. When setting up and installing a gateway in the field, a preliminary analysis of the ambient noise level is essential to prevent high noise or strong interference from impacting communication distance and reliability. While we aim for the lowest possible ambient noise, completely noise-free environments are unrealistic. An ambient noise floor of around -110dBm is generally considered ideal. A maximum noise level of -100dBm is also permissible; otherwise, communication distance will be severely limited. If the noise level exceeds -95dBm, it is strongly recommended to switch operating frequencies to avoid interference.

#### IoTSquare

IoTSquare([https://cloud.iotsquare.xyz](https://cloud.iotsquare.xyz/)) is a program used to connect to IoTSquare server. It is mainly composed of two parts, one is bridge and the other is Packet Forwarder.

The bridge is the core program used to connect to the IoTSquare server, while Packet Forwarder is used to send and receive LoRaWAN data. When Packet Forwarder is not started, it is only used for remote device management by the server. When Packet Forwarder is started, it can be used for both remote device management and receiving LoRaWAN data. Devices only have the bridge running by default. IoTSquare does not require configuring frequency plans on the gateway; users can select the appropriate frequency plan when registering the gateway with the IoTSquare server and then start the Packet Forwarder component.

![image-20230209161004622](https://risinghf-wiki.oss-cn-shenzhen.aliyuncs.com/upload/img/13c24de683d4f71808a2d61017b08ecf.png)

Gateway ID: Enter the gateway ID. IoTSquare uses the Ethernet MAC address with FFFE inserted in the middle as the gateway ID by default. Users can modify it according to their needs.

MQTT Server Address: Enter the target server's MQTT address and port number.

HTTP Server Address: Enter the HTTP address and port number of the destination server.

Start/Reboot: Start the bridge

Close: close the bridge. If the Packet Forwarder is also started, it will be closed together with the Packet Forwarder.

Start: start the LoRa service. If bridge is closed, start it together with bridge. If the device starts Other LoRa-related services, the system will shut down other LoRa services and start the services.

## Quick start

Let's use servers, gateways, and nodes to build a complete LoRaWAN communication example to quickly understand how to use the Gateway.

Prepare the environment:

*   LoRa server: [IoTSquare](https://cloud.iotsquare.xyz/)。
    
*   LoRa Gateway: RHF2S208D。
    
*   LoRa node: [RHF3M076B](https://wiki.risinghf.com/zh/01/01/05/01/#%E4%BA%A7%E5%93%81%E7%AE%80%E8%BF%B0) developed and produced by our company.。
    

In addition, a computer and a router connected to the Internet are required, and the RHF2S208D gateway and the computer are connected to the router to form a local area network (the gateway is connected by Ethernet). Then, log on to the router management interface on the computer to query the IP address of the Gateway so that you can log on to the Web management platform to operate the Gateway later.

### Log on to the server

Access the [IoTSquare](https://cloud.iotsquare.xyz/) server on the browser of your computer and log in. Subsequent registration of gateways and devices must be completed on the [IoTSquare](https://cloud.iotsquare.xyz/) server.

**Note: If you do not have an account, please communicate with our company for business and help you register an account. **

### Register a Gateway

On the server homepage, choose [Gateway Management]->[add gateway] to register a gateway.

## FAQ

Q1: the system time is incorrect, which is sometimes different from the local time.

A1: The system uses the UTC time zone by default, so it is sometimes different from the local time of the user. On the web page, choose system configuration> time zone management to select a local time zone.



Q2: The LoRa component displays “Not used".

A2: When the unused equipment leaves the factory, the LoRa module does not work by default, so it is normal to display "Not used. When a user starts a LoRa-related service, the LoRa component displays "normal". If the LoRa component displays an exception, contact the manufacturer in time.



Q3: Can I connect to multiple LoRaWAN servers at the same time?

A3: no. The LoRa module in the gateway is an exclusive resource. Only one LoRa-related service can be started at the same time.



Q4: Why does the background noise scanning function always fail?

A4: First, make sure that the gateway is connected to the board normally, and then make sure that the start frequency and end frequency of the scan are incorrectly filled. Of course, the scan range may be too large, resulting in timeout.



Q5: Why does one-click diagnosis succeed but still fail to receive LoRaWAN data from the node?

A5: One-click diagnosis only diagnoses the hardware related to the LoRa service. Successful diagnosis only means that the hardware is fine. However, LoRaWAN received data is also related to the configuration of servers, gateways, and nodes. We recommend that you check whether the frequency configuration and registration information of servers, gateways, and nodes are correct.



Q6: The Gateway has been registered to the LoRaWAN server, but why is the gateway not online on the server?

A6: Follow these steps to troubleshoot the problem:

1.  Check whether the network return is normal.
    
    - If wired is used, check whether the LED of RJ45 flickers. Flickering indicates that the connection is normal. If not, check whether the network cable or RJ45 interface and other hardware are normal.
    - If 4G is used, please check whether the 4G LED flickers. Flickering indicates that the 4G number is successfully dialed. If it does not flickers, please check whether the 4G antenna is connected, whether the SIM card is plugged in, whether the SIM card is overdue, whether the SIM card will be bound to the device, etc.
2.  Check whether the server address and port number are correctly filled in. Generally, this information can be obtained by consulting the user manual of the server or contacting the technical support of the server manufacturer.
3.  Check whether the network of the device accesses the server normally. In the [Network Configuration]->[Ping] function, enter the IP address of the server. If the ping fails, contact the network operator or server manufacturer to check whether the server cannot be accessed due to firewall problems. If you use 4G dialing, check whether the carrier has an IP whitelist mechanism.



Q7: Why is 4G dialing disconnected and redialed every few minutes?

A7: confirm the following two points:

1.  Check whether the dialed APN is correct with the network operator. If the error is incorrect, use the [network configuration]->[APN configuration] menu to delete the incorrect APN and enter the correct APN.
2.  Whether the operator has an IP address whitelist mechanism. If you have an IP address whitelist, add the server configuration-> IoTSquare-> MQTT Server address (for example, cloud.iotsquare.xyz) to the whitelist.



Q8: Why is there no data in the "Solar Battery Collection Data" menu?
